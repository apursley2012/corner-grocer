Corner Grocer Case Study

How I rebuilt and expanded a grocery-frequency application across C++, Python, SQLite, and the browser

Project Overview

The original Corner Grocer application was a C++ command-line program designed to analyze and report the frequency of grocery items sold in a store. It read daily sales data from a text file, stored and counted each item’s frequency using a map data structure, and displayed the results as either a full frequency list or an asterisk histogram.

The application solved a practical inventory-tracking problem by helping users identify purchasing trends for commonly purchased grocery items. While the original version was functional, it ran only as a standalone console program and had limitations in usability, structure, error handling, and long-term data storage.

Why I Chose This Project

I chose Corner Grocer because it provided a clear foundation for demonstrating a wide range of software development skills within one project. The original C++ version already had useful core logic, but it also left room for meaningful improvements in software design, algorithms and data structures, database integration, and user experience.

Using the same application throughout each stage also made it possible to show how I approach an existing codebase over time. Instead of replacing the original idea, I preserved its purpose and functionality while restructuring the code, adding practical features, and preparing it for future expansion.

The Original C++ Application

The original program used standard file streams and a std::map to store grocery item frequencies. Users could look up the frequency of one item, print the complete frequency list, or display a histogram showing each item’s count with asterisks.

The program completed its intended task, but the code was mostly procedural and lacked advanced error handling, modularity, persistence, and features beyond the original requirements. Item data existed only while the program was running, and the application had no way to save reports on demand or support more advanced sorting and searching.

Enhancement One: Rewriting the Application in Python

I rewrote the program from scratch in Python, incorporating modern design patterns, improved user feedback, and several structural and functional enhancements. Rewriting it in Python allowed me to adopt cleaner programming practices while preserving the original objectives and logic.

The enhanced version introduced a GroceryFrequency class that encapsulates the application’s core functionality, making the code more readable, testable, and maintainable. I replaced the C++ map logic with Python’s defaultdict, used context managers for file access, and separated the application into focused methods rather than keeping the logic in one procedural flow.

I also added input validation, exception handling, and clearer user messaging. Two new menu options allowed users to save a backup of the frequency list and a backup of the histogram at any point. The program confirms whether each file was successfully written and displays the full save path, which gives users clearer feedback and makes the backup features more useful.

Enhancement Two: Sorting, Searching, and Data Handling

I expanded the application’s data handling by adding alphabetical sorting, frequency-based sorting, and item search functionality. These features made the program more useful for realistic inventory tasks because users could organize the same data in different ways, quickly find a specific item, and identify the products sold most often.

I placed the sorting and searching logic into reusable helper functions rather than writing it directly inside the menu. This kept the main program focused on application flow and user interaction while keeping the data-processing logic modular, easier to test, and easier to maintain.

I also improved item normalization so capitalization and common singular or plural variations are treated as the same grocery item. For example, Apple, apple, and apples are processed as one item instead of creating separate records. Search also supports partial matches, which makes the feature more forgiving and practical for users who may not enter the exact stored name.

Enhancement Three: SQLite Database Integration

The next enhancement replaced the application’s in-memory data handling with a persistent SQLite database backend. In the earlier versions, data was loaded from a file and stored temporarily while the program was running. Moving the data into SQLite allowed the application to retain information across sessions without requiring a separate database server or additional setup.

I placed all database-related logic into a dedicated db_handler.py module. Database connections, table creation, inserts, updates, queries, and exports are separated from the menu and presentation logic. This separation of concerns makes the application easier to maintain and allows the database layer to be updated without rewriting unrelated parts of the program.

I used parameterized SQL queries, validated data before storage, and added exception handling around database operations. The enhancement preserved all existing non-database features, so the application gained persistent storage without losing the functionality already added during the earlier stages.

Creating the Static Browser Demo

The original C++ and Python versions were command-line applications, so I created a browser-based demo that could run directly through GitHub Pages. The demo keeps the same menu-driven purpose while presenting the features through a visual interface built with HTML, CSS, and JavaScript.

Because GitHub Pages cannot run Python or provide a live SQLite backend, I recreated the application logic in JavaScript and used browser storage to simulate persistence. The demo includes item lookup, full frequency reporting, a histogram, alphabetical and frequency-based sorting, partial search, item entry, and downloadable report exports.

This version is not intended to replace the original source code. It gives visitors a working way to interact with the project while the downloadable Python files preserve the full enhanced implementation.

Challenges and Decisions

Converting the project from C++ to Python required more than translating syntax. I had to decide how to restructure the original logic around Python’s design patterns, replace map behavior with a Python-appropriate data structure, and improve the application without changing what it was designed to do.

File writing and user error handling also required careful testing. I needed to make sure backup files were created successfully, permissions or missing files did not cause the program to fail without explanation, and all menu input was validated before it reached the application logic.

The database enhancement created a different challenge because it significantly changed the program’s internal data flow while the user-facing behavior needed to remain consistent. I had to map the existing file-based process to database operations and verify that all previous features continued to work exactly as expected.

For the static demo, the main tradeoff was recreating server-side behavior within the limits of a client-only GitHub Pages site. Browser storage provides a practical demonstration of persistence, but the Python and SQLite versions remain the complete application implementations.

Result

The final project is a cleaner, more extensible application that preserves the purpose of the original C++ program while improving its structure, reliability, and usefulness. The Python rewrite demonstrates language portability and modern software design. The sorting and search features make the data easier to work with. SQLite adds persistent storage, and the static browser demo makes the project accessible to visitors without requiring them to install or run the command-line application.

Throughout the project, I strengthened my skills in modular design, file management, exception handling, Python syntax, SQL, input validation, and interface design. More importantly, the project shows my ability to take existing logic and reimplement it in a way that is more organized, maintainable, and user-friendly without losing the original functionality.
